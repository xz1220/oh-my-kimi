# Verify

当用户希望确认某个功能、修复或重构真的能工作时，使用本 skill。

## 目标
把模糊的「应该能跑」变成具体的证据。

## 工作流
1. 明确必须被证明的具体行为。
2. 优先用已有测试。
3. 如果覆盖不足，跑最窄的直接验证命令。
4. 如果自动化还不够，描述手动验证步骤并收集可观察的具体证据。
5. 只报告真正被验证过的内容。

## 验证顺序
1. 已有测试
2. typecheck / build
3. 收窄的直接命令检查
4. 手动或交互式验证

## 规则
- 没有证据，不要说一处改动「完成了」。
- 检查失败时，明确把失败贴出来。
- 如果根本没有现实的验证路径，明说，不要虚张声势。
- 优先给简洁的证据摘要，而不是嘈杂的日志。

## 输出
- 验证了什么
- 跑了哪些命令 / 测试
- 哪些通过了
- 哪些失败或仍未验证
