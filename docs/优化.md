# 优化点记录

记录在阅读 / 分析 oh-my-kimi 仓库过程中发现的可优化项，按 skill / 模块分组。每条优化项包含：背景、问题、建议方案、优先级。

---

## writer-memory：CRUD 与语义操作分离（混合架构）

**优先级**：中

### 背景

`skills/writer-memory/` 当前是一份**纯 prompt-driven** 的 skill：

- 唯一被 Kimi CLI 真正注册的命令是 `/skill:writer-memory`。
- SKILL.md 里列出的 `init / char / rel / scene / theme / world / query / validate / synopsis / status / export / backup` 等子命令**没有任何代码实现**——它们由 LLM 读完 SKILL.md 后用通用工具（Read / Write / Edit / Bash）现场「演绎」出来。
- 上游 `Yeachan-Heo/oh-my-claudecode` 的 `lib/` 下有 5 个 TS 参考实现（`memory-manager.ts`、`character-tracker.ts` 等），但 SKILL.md 不引用它们、仓库内也无人 import，仅作为「给 Claude 读的精确化 spec」存在。
- 本地 fork 在汉化时把 `lib/` 和 `templates/` 清空了，连这份参考实现都没有。

### 问题

把 CRUD 也交给 LLM 演，带来一组确定性代价：

| 维度 | 现状 | 影响 |
|---|---|---|
| Token 成本 | 每次调用都把 SKILL.md（约 16 KB）+ memory.json 全文塞进上下文 | 高频小操作（`char list`、`status`）极不划算 |
| 原子性 | 备份 → 写入两步之间若中断，可能丢数据 | 长会话下偶发数据损坏 |
| Schema 一致性 | 不同会话生成的 JSON 字段顺序 / 嵌套形态可能漂移 | 后续 query 取不到字段 |
| 速度 | LLM 做 JSON read-modify-write 比脚本慢 1-2 个量级 | 批量 `rel update` / `scene add` 体验差 |
| 校验缺失 | 没有 schema validator 兜底 | 写坏的 JSON 直到下次读才暴露 |

而 `query / validate / synopsis` 这三个命令**注定需要 LLM 语义判断**，没法用纯代码实现：

- `validate 새랑 "사랑해..."`：要判断这句台词是否违反人设语调，纯函数做不了。
- `query "새랑은 이 상황에서 뭐라 할까?"`：自然语言推理，必须 LLM。
- `synopsis`：综合所有记忆生成情感导向叙述，必须 LLM。

### 建议方案：混合架构

```
真工具层（lib/ 编译为 CLI 或 MCP server）：
  - char add / update / list / get
  - rel add / update / list
  - scene add / list / get
  - theme add / list
  - world set / get
  - status / export / backup / init

Skill 演绎层（保留在 SKILL.md）：
  - query <自然语言>
  - validate <character> <dialogue>
  - synopsis
```

**实现路径**（两种可选）：

1. **CLI 包装**：把上游 5 个 TS 文件编译成 `writer-memory-cli` 可执行文件，SKILL.md 改为指导 LLM「`Bash` 调用 `writer-memory-cli char add ...`」。优点：零额外依赖、跨 harness 通用。缺点：需要分发预编译产物。
2. **MCP server**：把 CRUD 暴露成 MCP 工具，由 Kimi CLI 直接 function-call。优点：参数化更严谨、错误返回结构化。缺点：上游 Claude Code 之外的 harness 支持度不一。

### 收益

- **快路径**：`char list` 这种调用从「LLM 全文重写 JSON」降为「ms 级脚本执行」。
- **可靠性**：原子写、schema validator、备份保证都由代码而非 prompt 保证。
- **Token 节省**：高频 CRUD 不再消耗 LLM 上下文。
- **LLM 价值集中**：模型只在它**唯一能干**的事（语义校验、情感推理）上发力。

### 代价 / 权衡

- 失去「改一行 SKILL.md 立刻改行为」的极致灵活性。
- CRUD 层需要维护构建 / 发布流程（上游目前没有）。
- 跨 harness portability 略下降——目前 SKILL.md 在任何支持 SKILL.md 约定的 agent 上都能跑；引入二进制后需要分发对应平台的构建产物。

### 验收标准（如果实施）

- [ ] `lib/*.ts` 可独立编译为 CLI 或 MCP server。
- [ ] SKILL.md 子命令表标注每条命令的执行路径（tool / skill）。
- [ ] `validate / query / synopsis` 三条命令保持原有 LLM-driven 行为。
- [ ] 提供 schema 校验脚本，可在 CI 中跑通。
- [ ] 安装脚本 `scripts/install.sh` 同步处理新增的可执行文件落位。
